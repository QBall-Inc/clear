How CLEAR keeps an agent's memory honest.

A markdown memory for coding agents is the easy part. Keeping it from rotting — pruning what's no longer true as the code moves on — is the hard part. Here's the whole system, one layer at a time, in the order each piece depends on the last.

1The problem why agents drift on long projects

On anything bigger than a weekend project, a coding agent loses the plot. It starts every session cold, re-derives decisions you already made, and — the worst failure — confidently acts on guidance that stopped being true three commits ago. A memory that never forgets the wrong things isn't memory; it's a liability.

cold start
Session N+1 begins with none of session N's context. You re-explain the architecture every morning.
re-derivation
The same decisions get rediscovered (and sometimes reversed) because nothing recorded them.
stale guidance
Old patterns resurface after they've been superseded. A confidently wrong agent is worse than a forgetful one.
the fix starts with the atom of knowledge
2A concept is a file typed markdown, bound to code

The unit of knowledge is one markdown file with structured frontmatter. The frontmatter is the machine-read contract; the body is human detail. No database, no server — just diffable files in your repo, reviewed in the same PR as the code. The related_files list is what binds a concept to the code it concerns, so touch a bound file and its concepts surface automatically.

.clear/knowledge/TD-014.mdtechnical-decision
---
id:            TD-014
title:         Use cursor-based pagination for the public API
type:          technical-decision
status:        active
tags:          [api, pagination, performance]
related_files: [ src/api/list-handler.ts, src/api/cursor.ts ]
description:   Offset pagination broke under deep page loads; cursors are stable.
---
every concept is one of seven kinds
3Seven knowledge types closed set, on purpose

The type isn't a free-text tag — it's a closed, opinionated set, because a known type is what lets the system treat a decision differently from a process or a lesson. Code is the first domain, not the only one (people, orgs, and events are on the roadmap).

TD
Technical decision
e.g. "Adopt JWT over session cookies for the gateway" — the call, and the alternatives weighed.
PAT
Architectural pattern
e.g. "Repository pattern wraps every data source" — a reusable structure in the codebase.
LES
Lesson learned
e.g. "The MCP server dropped tools over 200 — paginate the registry" — a pitfall found the hard way.
BR
Business rule
e.g. "Refunds are blocked after 90 days" — a domain rule the software must honor.
IW
Institutional wiki
Reference knowledge sourced from a canonical doc — the RFC, the runbook, the spec.
PROC
Process
A recurring procedure: its trigger, frequency, and steps — e.g. the release checklist.
SH
Stakeholder
A person, team, service, or vendor, and what it owns — e.g. "Auth service · owned by Platform team."
the part a plain markdown wiki leaves out
4The lifecycle the dimension OKF scopes out · animated below

This is the heart. Every concept carries a status, and that is what keeps the graph fresh instead of rotting. When a newer decision lands, the old one is superseded (link recorded both ways). When something simply stops being true, it's deprecated. Neither is deleted — both are surfaced as stale, so an agent is never silently handed dead guidance.

pending
captured, provisional — awaiting review
active
confirmed, in force — eligible to surface
superseded
replaced by a newer concept — link both ways
deprecated
no longer true — kept, surfaced as stale
archived
removed from surfacing — terminal

but who writes and prunes all this? the loop does
5The loop that drives it plan → schedule → act → manage

You never maintain the knowledge base as a side chore. A structured development loop is the engine: the act phase captures decisions and lessons as you build; the manage phase, at session handoff, is what triggers supersession and pruning. Building the software maintains the graph.

  • plan — capture intent as phases CLEAR tracks across sessions.
  • schedule — break it into trackable workpackages.
  • act — build; decisions & lessons are captured here, bound to the files you touch.
  • manage — session handoff; the loop's progression supersedes & prunes.
  • the graph then serves the right context back into the next act.
how this lines up with google's new spec
6Where OKF fits convergence, not a priority claim

In June 2026 Google open-sourced the Open Knowledge Format — a draft spec for typed, resource-bound, markdown-native knowledge. It validates the model CLEAR has been running since 2025. The honest read is three layers: OKF specifies a format; CKS adds the lifecycle; CLEAR is the engine that lives it.

Layer 1 · format
OKF — Google, v0.1
Typed, described, tagged, resource-bound, cross-linked markdown. Status, supersession and pruning are explicitly out of scope.
Layer 2 · + lifecycle
CKS — the CLEAR spec
Format parity, plus opinionated types and the full lifecycle: status, supersession, deprecation surfacing, pruning, provenance.
Layer 3 · the engine
CLEAR — shipped 2025
Captures during development, serves the right context at the right time, syncs knowledge to project state. The system that runs the spec.
The graph gives structure to traverse. The lifecycle keeps it true. Building the software maintains both.
